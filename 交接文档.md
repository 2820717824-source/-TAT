## 一、项目概况

行业热点爬取工具。输入行业关键词，自动搜索多源热点、爬取全文、转 Markdown 存储。

### 当前状态

三批升级全部完成，项目从单文件 827 行 → 模块化架构（12 模块）。

```
├── hotspot_crawler.py    ← CLI 入口 + 降级路径
├── searcher.py           ← 搜索编排器（策略链模式 + 随机间隔）
├── sources/              ← 搜索源包
│   ├── __init__.py       ← SearchSource 基类 + @register_source + importlib 自动发现
│   │                       + fetch_url(指数退避重试) + clean_text(清洗工具)
│   ├── baidu.py          ← 百度热搜源
│   ├── zhihu.py          ← 知乎热榜源（支持 `--login zhihu` 自动缓存Cookie）
│   ├── bing.py           ← 必应搜索源
│   └── weibo.py          ← 微博热搜源
├── config.py             ← CrawlerConfig + YAML/CLI 合并 + 源名校验 + per-source 配置
├── fetcher.py            ← 双引擎获取器（requests + Playwright 智能降级）+ 内容质量检查
├── browser_fetcher.py    ← Playwright 渲染 + 隐身配置（批1）
├── dedupe.py             ← SHA256 去重 + SQLite 持久化
├── storage.py            ← 多格式输出（MD/JSONL/CSV）
├── engine.py             ← CrawlerEngine 全流程编排器（集成日志 + 失败恢复）
├── run_state.py          ← TaskLogger（三流JSONL日志）+ ResumeState（SHA256失败恢复）
├── cookie_manager.py     ← Cookie 持久化管理（保存/加载/验证各源Cookie）
├── requirements.txt      ← 项目依赖
└── docs/                 ← 文档、spec、plan
```

## 二、已完成功能详解

### 批次 1：JS 渲染 + 配置系统 ✅

**解决了什么：**
- 原爬虫只能爬静态页面，JS 渲染的站点拿不到内容
- 原来所有参数靠命令行传，没法固化

**怎么解决的：**
- `browser_fetcher.py`：Playwright 渲染 + 从 image_crawler 借鉴的隐身配置 5 件套（覆盖 webdriver/plugins/languages/chrome/permissions）
- `config.py`：YAML 配置 + CLI 合并，CLI 参数优先
- `fetcher.py`：双引擎智能降级（requests → Playwright），检测 SPA 特征自动切换

### 批次 2：工程化 ✅

**解决了什么：**
- 重复爬相同内容
- 只有 Markdown 一种输出格式
- 流程没有统一编排

**怎么解决的：**
- `dedupe.py`：SHA256(title+url) 去重，SQLite 持久化，跨运行有效
- `storage.py`：MD / JSONL / CSV 三种输出格式
- `engine.py`：CrawlerEngine 编排器，统一管理搜索→爬取→去重→存储→报告

### 批次 3：搜索源模块化 + 微博热搜 ✅

**解决了什么：**
- 原来 HotSearcher 类内聚了所有搜索源逻辑（~300 行），新增源必须改核心代码
- 没有微博热搜源
- 网络请求失败没有重试
- 文本内容没有清洗

**怎么解决的：**
- `sources/` 包：SearchSource 基类 + @register_source 装饰器注册机制 + importlib 自动扫描
- `searcher.py`：策略链模式编排多源搜索，每条源独立 try/except
- `sources/weibo.py`：微博热搜源（接口 + 备用降级）
- 指数退避重试 fetch_url()：只在 429/5xx 重试，401/403/404 直接返回
- clean_text() 清洗工具：HTML unescape、空白规范化
- 随机间隔 delay：`delay * (0.5 + random())` 代替固定 delay
- config.py 源名校验、per-source 配置（cookie 透传）

### 项目 3：python_spider — 教学示例 + 一个小型爬虫框架原型

**定位**：爬虫教学示例（urllib/requests/Selenium/Ajax/MongoDB）+ WeiXin/ 模块是一套完整的小型爬虫框架
**质量**：教学级为主，但 WeiXin/ 模块质量中等偏上

#### 可以直接用的代码

无。但 WeiXin/ 模块的设计分层可以直接复制模式：

```
config.py → 配置层（Redis/MySQL/代理配置）
request.py → 请求模型层（WeixinRequest：请求 + callback + need_proxy + fail_time 打包）
db.py      → 持久化队列层（Redis FIFO，pickle 序列化请求对象）
mysql.py   → 数据存储层（通用 dict → INSERT 工具，带参数化查询防 SQL 注入）
spider.py  → 引擎调度层（Session + 代理轮换 + 回调驱动调度）
run.py     → 入口层
```

关键代码点：
- **spider.py line 30-42 代理轮换**：从外部 API（`http://127.0.0.1:5555/random`）拉取代理，按请求粒度注入
- **spider.py line 121-143 调度循环**：Redis 队列 pop → 执行请求 → callback 处理 → 结果入队/入库，类似 Scrapy Engine 架构
- **spider.py line 110-119 失败重试**：MAX_FAILED_TIME=20 + Redis 持久化，支持断点续爬
- **request.py line 5-11**：将请求和回调处理逻辑打包为独立对象

#### 需要适配才能用的代码

| 代码 | 文件位置 | 说明 | 适配难度 |
|------|---------|------|---------|
| **今日头条 Ajax 逆向** | `Toutiao/分析Ajax来爬取今日头条街拍美图.py:19-42` | 分析 API 参数（offset/count/cur_tab），直接调 JSON 接口 | 后续添加头条源时参考 |

#### 重要发现

- **WeiXin/** 是整个仓库中最接近工程化的模块。分层设计使得每一层责任单一、可替换（db.py 的 Redis 可换成 Kafka，mysql.py 可换成 SQLAlchemy）。spider.py 同时承担了引擎和调度两个职责，可再拆出一个 scheduler.py
- **Ajax 逆向思路**有价值：有些站点直接调 API 比渲染页面更高效。Toutiao/ 里从 `<script>` 标签用正则提取 `gallery: JSON.parse(...)` 内嵌 JSON 的手法——绕过渲染直接拿数据——对 SPA 站点很有参考价值

### 项目 4：javaCrawling — 最有架构启发（Java）

**定位**：Spring Boot + WebMagic 的企业级爬虫管理平台
**质量**：过时（Spring Boot 1.4, WebMagic 0.7.x），但架构设计最有借鉴价值

#### 可以直接用的代码

无。Java 代码不能直接用在 Python 项目里。

#### 需要适配才能用的代码

| 代码 | 文件位置 | 说明 | 适配难度 |
|------|---------|------|---------|
| **动态代理池** | `NetPoxyService.java:1-86` | data5u API 取 IP → TTL 过滤 → 随机轮换。约 40 行 Python | 中 |
| **RinseRule 清洗 DSL** | `RinseHtmlUtil.java:1-320` | 用 JSON 规则描述 HTML 操作（replace/delete/add/copy），Python 可用 bs4 实现 | 中 |
| **Redis 三队列优先级调度** | `RedisQuenUniqPriorityScheduler.java:1-208` | zset(高优) + lpush(普通) + zset(低优) + sadd 去重 | 低 |
| **混合下载器策略** | `WeChatSeleniumDownloader.java:1-411` | URL 模式路由：微信 URL → Selenium，其他 → HttpClient | 中 |

#### 重要发现

- **配置驱动的爬虫编排是最大的架构启发**。WebmagicConfig.java 是配置根，processor/downloader/pipeline 全部通过 Spring Bean Name 字符串实现动态加载。相当于用 JSON 描述一个爬虫的全部行为，运行时动态组装。这个思路可以直接移植到我们的 engine.py 中
- **Pipeline 责任链设计成熟**：清洗(RinseComm) → 图片处理(RinseWeChat) → 入库(MySqlArticle) → 异常处理(ExceptionPipeline)，顺序在配置中可插拔
- **NetPoxyService 的 data5u 代理**在 Python 中对应快代理/芝麻代理。可以设计一个 ProxyProvider 抽象类，让不同代理源可插拔。500 次重试的"永不放弃"语义需要改为循环而非递归（原代码递归可能栈溢出）
- **三个下载器之间严重 copy-paste**（handleResponse/getContent/getHtmlCharset/selectRequestMethod 在 WeChatSeleniumDownloader/WeChatDownloader/DynamicProxyHttpClientDownloader 中几乎逐行相同）。我们的模块设计要避免这种重复

## 十、给下个 Agent 的建议

1. **第一优先**：动态代理池 proxy_manager.py（~60行，参考 javaCrawling NetPoxyService）。目前百度/微博经常返回 0 条，可能是因为 IP 被限
2. **第二优先**：detail 详情页提取加 keep_on_detail_failed 回落（~50行，来自 universal-crawler）

7. **注意**：universal-crawler 的 auth 6 种类型 0 种实现，不要被 YAML 配置迷惑
